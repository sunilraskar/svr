The ETL Design Principles

The ETL framework would be a centerpiece to implement the business data strategies. It will be essentially next generation ETL framework which would pull data from various sources (structure/unstructured data), reshaping it, applying business rules, loading to the appropriate destinations, and validating the results using analytics and dashboard. The modern framework must have a number core architectural principles Which includes -

Any solution designed and developed using this new framework should not be a patch work of legacy and modern systems, abandoned and duplicate work which cause setback in business

The ETL framework must be first class high speed in loading volume data — Companies have been investing significantly in building next-generation platforms and products using new technologies. So, it is at most import that framework delivery high performance otherwise it led to disruption in services and can profoundly impact the brand image, customer loyalty, and revenue.

The framework must be highly Scalable — There should be a single point of failure. The Framework must able install on more than one node so that there is no down time. It should allow increase or decrease (Scale Up/Scale Down) the concurrent Data movement at each node job.

It is a well-known fact that any standardization positively impacts the business. Similarly, the framework must enforce writing same code with consistent standards and the standard must be well established in the industry so that supporting the code is not nightmare which will ensure smooth functioning of software programs and is vital for the success of most organizations.

The framework dataload state must not have scheduling problems which lead to inefficiency by either not producing high-quality results. It should allow data state scheduling which help efficiently large designs with complex flow with simple design.

The framework error Handling must be robust. If a program cannot successfully do what it is designed to do an exception should be thrown and systems should maintain the state so that users can work from the point of failure onward which will help organizations to develop ELT solutions in budget and time.

Essentially, ETL framework should provide organizations with the same outcome when implemented through out company.

Overview of the ETL framework Components.

The framework must have following four type of components.

Connect and collect

Enterprises have data of various types that are in disparate sources on-premises, in the cloud, structured, unstructured, and semi-structured, all arriving at different intervals and speeds. Therefore, the framework must have built in components to design the first step in the ETL solution.

The framework allows connect to all the required sources of data and processing, such as software-as-a-service (SaaS) services, databases, file shares, and FTP web services.

The next step in an ETL process is to move the data as needed to a centralized location for subsequent processing.

Without the ETL framework, enterprises build custom data movement components or write custom services to integrate these data sources and processing which is expensive and hard to integrate and maintain such systems. In addition, they often lack the enterprise-grade monitoring, alerting, and the controls that a fully managed service can offer.

With the ETL framework, you have components like Copy Activity in a data pipeline to move data from both on-premises and cloud source data stores to a centralization data store in the cloud for further analysis. For example, in Data factory you can collect data in Azure Data Lake Store and transform the data later by using an Azure Data Lake Analytics compute service.

Transform and enrich

After data is present in a centralized data store in the cloud, you need to components to reliable transformations the data to produce transformed data on a maintainable and controlled schedule to feed production environments with trusted data.

Publish

After the raw data has been refined you need components so publish it so that data is business-ready consumable form and available in popular data warehouse such as Azure Data Warehouse, Azure SQL Database, Azure Cosmos DB, or whichever analytics engine your business users can point to from their business intelligence tools.

Monitor

After you have successfully built and deployed your data integration pipeline, providing business value from refined data, monitoring the scheduled activities is so important. So, the framework must have built-in components which support pipeline monitoring via Azure Monitor, API, PowerShell, Log Analytics, and health panels on the Azure portal.

Stages of Data Flow Using Azure Data Factory

The below diagram demonstrates the overall Stages of the Data flow Job that can be implemented using the Azure Platform:

1. Input phase: The objective of this phase is to load data that will be processed by the Dataload Job. This is typically achieved using The NextGenDataload Connectors, which should be pre-build connector for example text file, SQL Database, App Connector. The data is then loaded onto a persistent disk to ensure processing if a failure occurs.

2. Load and dispatch phase: This phase is implicit (i.e. batch process developers do not have access to this phase). The objective of this phase is to divide loaded data from the input phase into individual records, then distribute it for further processing via a persistent message queue.

3. Process phase: This phase consists of one or more batch steps. A batch step is a sequence of message processors that individually process each record in the batch from the persistent queue — loaded during the load and dispatch phase. Steps in this phase include transforming data before updating, inserting data into another application, enriching data via API calls, etc.

4. Completion phase: This is the final phase, and it is typically used to provide a summary report on the batch process. Teams can also use this phase to perform post-processing steps, such as cleaning up data, sending notifications to initiate new processes, etc.

Conclusion

The reality is that most organizations have a wide variety of modern and legacy systems that hold valuable data. To tap into that information and further data-driven decision making, many organizations turn to ETL solutions; however, many organizations are facing setbacks because traditional approaches to implementing ETL solution create a patch work of legacy and modern systems, abandoned code, and duplicate work. Therefore, some organizations are turning to a new ELT platform and Organization are moving to Azure Data Factory, Mule Soft or Alteryx to implement these capabilities.
